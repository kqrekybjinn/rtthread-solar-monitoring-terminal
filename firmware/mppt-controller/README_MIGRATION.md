# STM32G474 RT-Thread MPPT Port

This directory is the standalone RT-Thread/SCons project for the PCB02 MPPT controller port.

## Project Type

- MCU: STM32G474RET6
- RTOS: RT-Thread
- Build system: SCons
- Source baseline: `https://gitee.com/muyan3000/esp32-mppt/tree/master/esp32_buck_boost/firmware`.
  The published firmware directory mainly contains release binaries; the readable control source is in
  `ARDUINO_MPPT_FIRMWARE_V2.1` in the upstream repository.
- Main application directory: `applications/pcb02`

This is not an `rtthread-cmake` project. It is the current working RT-Thread BSP project copied from the tested RT-Thread Studio workspace and made self-contained under this repository.

## Important Directories

- `applications/pcb02`: STM32G474 MPPT application modules.
- `board/CubeMX_Config`: STM32G474RET6 CubeMX pin and peripheral configuration reference.
- `libraries/HAL_Drivers`: RT-Thread STM32 HAL driver glue.
- `packages/stm32g4_hal_driver-latest`: STM32G4 HAL package.
- `rt-thread`: RT-Thread kernel and components.

## PCB02 Module Split

- `pcb02_app`: RT-Thread task entry, periodic service call and low-rate status log.
- `pcb02_service`: external control API for future CAN/UART/RK3506 control.
- `pcb02_sensors`: INA226 and ADC aggregation.
- `pcb02_control`: MPPT, CC/CV and protection logic.
- `pcb02_hw`: GPIO/PWM safe state and output application.
- `pcb02_shell`: MSH test commands.
- `pcb02_ina226`: INA226 I2C driver.
- `pcb02_adc`: NTC ADC driver.
- `pcb02_config`: default runtime parameters and pin mapping.

## Build

### VSCode + RT-Thread Extension

This workspace has been initialized for the RT-Thread VSCode extension:

- `.vscode/project.json` is generated by `scons --target=vsc -s` and drives the RT-Thread "Project Files" tree.
- `.vscode/tasks.json` provides `RT-Thread: build`, `RT-Thread: clean`, `RT-Thread: rebuild`, `RT-Thread: menuconfig`, and `ST-Link: flash bin`.
- The build task uses Python 3 plus SCons and the RT-Thread Studio ARM GCC toolchain paths configured in `.vscode/settings.json`.

If the RT-Thread side panel still shows `No .vscode/project.json found`, click refresh in the RT-Thread view or reload the VSCode window.

Set the toolchain path, then build with SCons:

```bat
set PYTHONUTF8=1
set RTT_ROOT=%CD%\rt-thread
set RTT_CC=gcc
set RTT_EXEC_PATH=E:\toolchain\RT-Thread\RT-ThreadStudio\platform\env_released\env\tools\gnu_gcc\arm_gcc\mingw\bin
set PATH=C:\Users\Amu\.cache\codex-runtimes\codex-primary-runtime\dependencies\python;C:\Users\Amu\.cache\codex-runtimes\codex-primary-runtime\dependencies\python\Scripts;%RTT_EXEC_PATH%;%PATH%
scons -j16
```

Expected output files after a successful build:

- `rt-thread.elf`
- `rt-thread.map`
- `rtthread.bin`

## MSH Test Commands

The CAN/FDCAN driver is intentionally not enabled yet. Use RT-Thread MSH to test the service layer first.
The commands are split into three layers:

### Hardware Bring-Up

```text
pcb02 status
pcb02 safe
pcb02 log <period_ms> <count>
pcb02 fan <0|1>
pcb02 pwmtest <permille>
pcb02 clear_fault
pcb02_ina_probe
pcb02_i2c_scan
```

`pwmtest` is open-loop PWM and is only for oscilloscope and gate-driver validation. Stop it with
`pcb02 safe`.

### Closed-Loop Power-Stage Tests

```text
pcb02 cvctrl <fastpi|pi|inc|fuzzy|adapt> <target_v> <iout_limit> <max_permille> <kp> <ki_per_s> [kd] [slew_v_s=0]
pcb02 mppttest <po|inccond|vspo|adapt> <vout_max> <iout_limit> <max_permille> [step_counts]
pcb02 chgtest <vbat_cv> <ichg_limit> <vin_min> <max_permille>
```

`cvctrl` is a bench supply style constant-voltage test. It is useful for checking compensation and
power-stage stability, but it is not the final solar-charging mode.

`mppttest` perturbs PWM while watching input power, with output-voltage and output-current limits.
It is for MPPT algorithm bring-up against an electronic load or PV simulator.

`chgtest` is the current charging-stage test interface. It follows the upstream firmware idea:
real-time input-voltage feedforward, output current limit, output CV limit, PV undervoltage throttle,
and bypass/backflow enable only when input is higher than output by the configured dropout.

### Configuration And Future App Control

```text
pcb02 status
pcb02 enable 1
pcb02 enable 0
pcb02 mode mppt
pcb02 mode cv
pcb02 output charger
pcb02 output supply
pcb02 limits 12.6 5.0
pcb02 clear_fault
pcb02 safe
```

Future CAN support should parse CAN frames and call the same `pcb02_service_*` APIs used by these MSH commands.

## Upstream Functional Baseline

The STM32 port must eventually cover the work already present in the original ESP32 firmware:

- Predictive PWM startup from `Vout / Vin`, constrained by a maximum duty limit.
- CC/CV charger mode: reduce PWM on output overcurrent or output overvoltage; increase PWM while below CV.
- MPPT charger mode: perturb-and-observe style tracking using input power, with voltage/current protection.
- Backflow/bypass control: enable only when input voltage is above output voltage plus dropout.
- Protection bits: overtemperature, input overcurrent, output overcurrent, output overvoltage,
  fatal low voltage, input undervoltage, battery-not-connected, and recovery request.

Current module testing is intentionally narrower than the final app. The `chgtest` command is the first
bridge from isolated buck testing toward the original charger behavior.

## Current Limitation

RT-Thread generic FDCAN is disabled because the BSP `drv_fdcan.c` API does not match the currently copied STM32G4 HAL package. The PA11/PA12 FDCAN1 pin plan remains in CubeMX. The recommended next step is a small PCB02-specific FDCAN adapter instead of patching the generic BSP driver.
